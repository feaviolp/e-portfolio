Agent Based Systems - Initial Post

Before discussing the advantages and disadvantages of agent communication languages (ACLs), I will first briefly discuss the difference between intelligent agents and objects to give a little context, because superficially they appear to be similar.

Odell (2002) gives a nice comparison, with the key feature that differentiates agents from objects being autonomy, because agents can react to their environment as well as instructions. This gives agents the ability to “say ‘No’ – as well as ‘Go.’” (Odell, 2002: 42). It is that autonomy that necessitates ACLs rather than typical method or function invocation, because when a method is invoked a particular response or action is typically expected. With an agent it’s more like a negotiation whereby information is exchanged, which the receiving agent adds to its own knowledge, including information based upon its environment, and then decides what to do.

I think it’s now clear why a different communication language is needed for agents. Knowledge Query and Manipulation Language (KQML) allows “two or more intelligent systems to share knowledge in support of cooperative problem solving” (Finin et all, 1994:93). The information is shared “independent of content syntax and applicable ontology” (Addakiri & Bahaj, 2013: 4329). So with KQML, agents with different goals and ontologies can share information as they each try to further their own goals.

It should be noted that not all agent-based systems are as heterogenous as this. Finin et al (1994: 7-10) discuss three types of multi-agent system architecture:

1.    Simple point-to-point connectivity, a static architecture and synchronous communication.

2.    Point-to-point connectivity with some multicasting, a static architecture and a mix of synchronous and asynchronous communication.

3.    Point-to-point, multicast and broadcast connectivity, a dynamic architecture and mainly asynchronous communication.

Whilst the context was using KQML for all three architectures, the first architecture could fairly easily use standard method invocation and still meet the brief of an agent-based system.

In complex multi-agent systems akin to the third architecture, however, an agent might not even know which agent it needs help from. A KQML message can be broadcast to agents to see who can help. Also facilitator agents can be used, whose job it is to find an agent able to help (Addakiri & Bahaj, 2013). Method invocation simply would not work in that scenario.

Even with all of the advantages of ACLs such as KQML, there are some disadvantages. Hai-long & Wu (2001) describe a deadlock situation when two agents try to start a communication with each other and neither recede, forcing both into a waiting state. Also, an agent who receives a KQML message isn’t obliged to respond or even acknowledge the message (Brooks et al, 1998), which can lead to sequencing problems in asynchronous solutions. So the power of the abstracted asynchronous communication enabled by KQML can also create potential problems. It seems that these could be avoided with robust design and test practices, but they are nonetheless challenges that would not be encountered in a simpler program with a single code-base synchronously invoking methods and functions.
